1. 产品背景与目标
1.1 背景
OpenClaw 是一个允许用户通过自然语言调用各种技能（Skill）的平台。开发者可以封装特定功能为 Skill，供 OpenClaw 用户安装使用。当前缺乏一个集中的 Skill 分发和商业化变现渠道。

1.2 目标
构建一个 Skill 市场三方中转站，实现以下核心能力：

允许平台方自主封装 Skill 并上架到市场。

OpenClaw 用户可以发现、购买并按安装次数付费使用 Skill。

平台提供完整的用户管理、计费、支付和统计分析功能。

系统需支撑高并发调用（预期数千 QPS），保证计费准确、响应迅速。

1.3 业务价值
为 Skill 开发者（平台方）提供变现渠道。

降低用户获取高质量 Skill 的门槛。

形成 Skill 生态闭环。

2. 用户角色
角色	描述	核心需求
普通用户	使用 OpenClaw 的终端用户，需要安装付费 Skill	浏览 Skill、购买次数包、查看余额和调用记录、管理 API Key
平台管理员	运营人员/开发者，负责 Skill 的上架和管理	管理 Skill（上架/下架/定价）、查看订单、管理用户、统计数据、对账处理
OpenClaw Skill	技术组件，非人类角色，负责调用计费 API	需遵循规范，携带 API Key 调用计费接口
3. 产品范围
3.1 包含功能
用户端（Web）：Skill 市场展示、购买次数包、个人中心（API Key 管理、余额、调用记录）。

管理后台（Web）：Skill 管理、订单管理、用户管理、财务统计、对账工具。

API 服务：用户认证、支付创建、支付回调、计费扣减、Skill 分发。

OpenClaw Skill 接入规范：定义 Skill 如何与计费系统对接。

3.2 不包含功能
OpenClaw 核心能力的修改（假设 OpenClaw 已支持安装外部 Skill 和配置 API Key）。

多语言支持（初期仅中文）。

退款流程自动化（初期手动处理）。

4. 功能需求
4.1 用户端（前台）
4.1.1 用户注册/登录
支持邮箱注册/登录，通过验证码或密码。

支持第三方登录（GitHub/微信扫码），简化流程。

注册成功后自动生成 API Key，可在个人中心查看。

4.1.2 Skill 市场
展示所有已上架的 Skill 卡片，包含：图标、名称、简短描述、价格（元/次）。

支持按分类、热度筛选。

点击 Skill 进入详情页，展示：详细描述、使用说明、定价方案（次数包列表）、“安装”按钮。

安装流程：

点击“安装”后，系统生成并返回 Skill 的下载链接（临时有效）和安装指引。

用户需复制 API Key 到 OpenClaw 的 Skill 配置中。

4.1.3 购买次数包
Skill 详情页展示可购买的次数包（例如：100次/50元，500次/200元）。

点击“立即购买”进入订单确认页，选择支付方式（微信/支付宝）。

调起支付（扫码或唤起支付 App）。

支付成功后，用户余额自动增加，并跳转到“购买成功”页。

4.1.4 个人中心
API Key 管理：展示当前 API Key，支持“重置”（生成新 Key，旧 Key 立即失效）。

余额明细：显示当前剩余次数（总余额），以及最近 30 条余额变动记录（时间、变动数量、原因、剩余）。

调用记录：分页展示历史调用记录：Skill 名称、调用时间、消耗次数、状态。

订单记录：展示历史订单：订单号、金额、购买次数、支付方式、支付状态、支付时间。

4.2 OpenClaw Skill 接入规范
4.2.1 Skill 包结构
符合 OpenClaw 官方规范，包含 plugin.json 和主入口文件（如 index.js）。

在 plugin.json 中声明 Skill 名称、版本、所需权限（若有）。

4.2.2 计费集成要求
Skill 代码中必须在执行核心逻辑前调用平台计费 API。

需从用户配置中读取 apiKey（OpenClaw 允许用户在安装后配置自定义字段）。

每次调用需生成唯一 invocationId（可使用 OpenClaw 提供的上下文 ID 或自己生成 UUID）。

计费 API 调用成功后，再执行实际功能；若失败，返回错误信息并引导用户续费。

4.2.3 计费 API 定义
接口地址：POST /api/billing/consume

请求头：Content-Type: application/json

请求体：

json
{
  "apiKey": "string",         // 用户 API Key
  "skillId": "string",         // Skill 唯一标识
  "invocationId": "string"     // 本次调用唯一 ID，用于幂等
}
响应：

成功 (200)：

json
{
  "success": true,
  "remaining": 99,           // 剩余次数
  "message": "扣费成功"
}
余额不足 (402)：

json
{
  "success": false,
  "error": "余额不足",
  "buyLink": "https://.../buy?skillId=xxx"  // 购买链接
}
其他错误 (4xx/5xx)：按标准 HTTP 状态码返回错误信息。

4.3 计费与支付
4.3.1 计费规则
按次计费，每次调用消耗 1 次余额（可配置不同 Skill 消耗不同次数，初期固定为 1）。

用户需先购买次数包，获得余额。

余额为整数，精确到 1 次。

调用时实时扣减，若余额不足则拒绝执行。

4.3.2 实时扣减与高性能要求
计费系统需支撑高并发（数千 QPS），采用 缓存加速 机制（如 Redis）进行实时余额操作。

扣减操作必须原子化，保证不出现超扣。

扣减成功后立即返回结果，调用流水异步写入数据库，避免数据库成为性能瓶颈。

系统需提供幂等能力，通过 invocationId 防止同一调用重复扣费。

4.3.3 支付集成
支持 微信支付（JSAPI 和 Native）和 支付宝（电脑网站支付 + 手机网站支付）。

用户下单时生成商户订单号，记录订单信息，状态为待支付。

调起支付后，用户完成支付，支付平台异步通知平台后端。

后端验证签名并更新订单状态为已支付，同时增加用户余额（实时更新缓存和数据库）。

支付成功页面提供“返回市场”和“查看余额”按钮。

4.3.4 对账与退款
每日生成对账单（支付平台侧 vs 平台订单），供财务核对。

退款流程：用户联系客服，管理员在后台发起退款（调用支付平台退款接口），退款成功后手动扣减用户余额或标记订单为已退款。

系统需支持缓存与数据库的一致性对账：每日比对缓存余额与数据库流水，确保数据一致。

4.4 管理后台
4.4.1 仪表盘
展示关键数据：今日订单数、今日收入、总用户数、总调用次数。

图表：近7日订单趋势、调用趋势。

4.4.2 Skill 管理
Skill 列表：展示所有 Skill，支持搜索、筛选（上架/下架）。

新增/编辑 Skill：

基本信息：名称、描述、图标（上传）、分类。

定价：设置单次价格（元/次），以及可购买的次数包（支持多个套餐）。

上传 Skill 文件（zip 包），系统存储到 OSS 并生成下载链接。

状态：上架/下架开关。

下架 Skill：已安装用户仍可继续使用，但新用户无法购买。

4.4.3 订单管理
订单列表：展示所有订单（订单号、用户、金额、次数、支付方式、支付状态、支付时间）。

支持按订单号、用户、状态筛选。

查看订单详情，支持手动标记退款（并调用支付平台退款）。

4.4.4 用户管理
用户列表：展示所有注册用户（邮箱、API Key、余额、注册时间）。

支持按邮箱搜索。

点击用户可查看详情：基本信息、余额变动明细、调用记录、订单记录。

支持管理员手动调整用户余额（加/减），并记录操作日志。

4.4.5 财务统计
统计总收入、各支付方式收入、各 Skill 收入排行。

导出报表（Excel/CSV）。

4.4.6 对账工具
提供缓存与数据库余额对比工具，支持手动触发对账任务，发现不一致时以数据库为准修复缓存。

展示最近对账结果及差异记录。

4.4.7 系统设置
支付参数配置：微信支付商户号、API 密钥；支付宝应用 ID、私钥等。

OSS 配置：AccessKey、Bucket、域名。

邮件服务配置（用于发送通知）。

缓存配置（如 Redis 连接信息，一般由环境变量管理，后台只读展示状态）。

5. 非功能需求
需求项	指标要求
并发能力	核心计费 API 支持 5000 QPS 以上（配合缓存）
响应时间	计费 API 95% 响应 < 100ms（含缓存操作）
数据一致性	最终一致性，极少情况下允许重复扣费（概率低于 0.001%），需有对账机制
幂等性	同一 invocationId 多次调用只计费一次
安全性	API Key 传输使用 HTTPS；支付回调验签；防 SQL 注入；防重放攻击；限制单用户调用频率
可用性	核心服务 99.9% 可用
备份	数据库每日自动备份，保留 7 天；Redis 数据持久化配置 RDB/AOF

7. 业务流程
7.1 用户购买次数包流程
graph TD
    A[用户选择次数包] --> B[创建订单(状态pending)]
    B --> C[调起支付]
    C --> D{用户支付}
    D -->|成功| E[支付平台异步通知]
    D -->|失败/取消| F[订单取消]
    E --> G[验签并更新订单为paid]
    G --> H[增加用户余额（缓存+数据库）]
    H --> I[发送支付成功通知]
    I --> J[用户查看余额更新]
7.2 Skill 调用计费流程（高性能版）
graph TD
    A[OpenClaw调用Skill] --> B[Skill读取apiKey和invocationId]
    B --> C[请求计费API /consume]
    C --> D{缓存余额检查}
    D -->|余额充足| E[原子扣减缓存余额]
    E --> F[设置invocationId幂等标记]
    F --> G[返回成功及新余额]
    G --> H[异步写入调用流水到队列]
    H --> I[队列消费者批量写入数据库]
    D -->|余额不足| J[返回402错误及购买链接]
7.3 每日对账流程
定时任务扫描所有活跃用户，读取 Redis 中的余额。

根据数据库 Invocation 流水计算用户应有余额。

对比两者，若不一致：

记录差异日志。

以数据库流水为准，更新 Redis 余额（并发送告警通知管理员）。

8. 验收标准
8.1 功能验收
用户可以注册、登录、查看 API Key。

用户可以在市场浏览并购买次数包，成功支付后余额增加（实时生效）。

用户可以在个人中心查看余额变动和调用记录。

管理员可以上架/下架 Skill，上传 Skill 文件。

管理员可以查看订单、手动调整用户余额。

OpenClaw Skill 调用计费 API 能正确扣费，余额不足时返回 402 和购买链接。

支付回调能正确更新订单状态和用户余额。

8.2 性能验收
计费 API 在 5000 QPS 压力下，平均响应时间 < 100ms，错误率 < 0.1%。

数据库连接池稳定，队列积压处理及时（消息延迟 < 1 秒）。

Redis 内存使用在合理范围，命中率 > 99%。

8.3 数据一致性验收
随机抽取用户，手工比对缓存余额与数据库流水计算余额，误差为 0。

模拟网络异常、服务重启等场景，验证幂等性生效，无重复扣费。

8.4 安全验收
API Key 在传输中使用 HTTPS。

支付回调验签正确，防止伪造通知。

用户余额调整操作记录日志，可追溯。

限流策略有效，超出阈值的请求被正确拒绝。
